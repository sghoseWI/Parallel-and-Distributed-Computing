    Important Logistical Notes for Working in AWS EMR:

    • You MUST set your IP address as the only one that can access the ports you open for this assignment. Hacking of AWS EMR clusters has been rampant and immediate, and will prevent you from working on the cluster (and will you get you warning emails from Amazon).
    • Jupyter notebooks are saved by default on the name node, which is ephemeral storage and will be gone when you close your cluster. For this reason, you MUST download your notebooks between cluster launches/shutdown. You can do this by selecting 'Download As' and using the `.ipynb` format, which you can then easily upload back into JupyterHub in your next cluster.

     

    Task 1. EMR Cluster and Spark Configuration In this assignment, you must launch an EMR cluster with Spark (just like the last assignment), selecting reasonable instance sizes (in terms of memory and processors). Based on these instances, configure (at least) spark. executor.instances, spark.executors.cores, and spark.executor.memory in such a way that makes sense for the data you are working with. These configurations must differ from the prior assignment. You must describe the instances used and clearly explain your thinking process in order to get credit.

     

    Task 2. Spark Machine Learning

    Load and join some of the LODES data and/or driving times dataset (see data documentation below). Next, develop a machine learning model that predicts an informative metric (this can be classification or regression).

    Specifically, implement the following in PySpark:

    • Split data into holdout and training sets.
    • On the training data, create a machine learning workflow that:
      • uses feature engineering with at least two transformers, building a column of feature vectors;
      • runs at least three types of machine learning models;
      • employs a parameter grid containing two or more hyper-parameters;
      • runs cross-validation - from which you should evaluate model performance and choose one final model;
    • Test model on holdout set and report its performance.
    • Persist your final model to S3.

    Please submit your finalized .ipynb and .html files for your analytical notebook.

     

    Data Documentation

    There are two distinct data sources available for this assignment, that include data on commuting and employment (by census block) and travel distances by car (by census tract). All of these datasets contain Census Tract identifiers, since the first 11 digits of a census block are the tract identifier.

    LEHD Origin-Destination Employment Statistics (LODES)

    The definition of variable codes, datasets, etc. can be found at the latest LODES 7.3 Technical Documentation. All LEHD Origin-Destination Employment Statistics (LODES) data are available, as described in the LODES documentation above. No changes have been made to the original CSV files. Data are available from 2002 to 2015. See the documentation above for caveats.

    For od (Origin-Destination) data, rac (Resident Area Characteristic) and wac (Workplace Area Characteristic) data, use the following URLs

    • s3://lsdm-emr-util/lsdm-data/lodes/od/od.parquet (14.3GB) 
    • s3://lsdm-emr-util/lsdm-data/lodes/rac/rac.parquet (2GB)
    • s3://lsdm-emr-util/lsdm-data/lodes/wac/wac.parquet (787MB)

    The LODES data is at the Census Block level, which is more granular (a small geography) than Census Tracks. However, you can get the Census Tracts by looking at the first 11 digits in the Census Blocks. 

    od = od.withColumn('w_tract', substring('w_geocode', 1, 11)) \
           .withColumn('h_tract', substring('h_geocode', 1, 11))

    rac = rac.withColumn('h_tract', rac_raw.h_geocode.substr(0,11))
    wac = wac.withColumn('w_tract', wac_raw.w_geocode.substr(0,11))

    Please note: Once you have the tracts, as seen above, you can aggregate each dataset (wac, rac, od) individually to the tract and year level (summing or averaging the other variables) then merge them together OR merge them together at the census block and year level (_geocode) and then aggregate them. You cannot merge them together on the tract level without first doing an aggregate, as this would be a huge many-to-many merge.

    You can then merge to the origin-destination data at the census tract level. 

     

    Driving Times and Distances Dataset

    Full Documentation: Census tracts are 2010 vintage, and the columns are the origin tract, destination travel, travel time in minutes, and travel distance in miles. These data were calculated by the Data Science team at the Urban Institute. 

    We use 2010 Census Tract geographies and select all unique combinations of Census tract pairs in the 50 states and District of Columbia that are located within 150 miles of each other by straight line distance, using the centroid of each Census tract for the “address” of that tract, which results in over 154 million pairs. We generate a routing engine from the Open Street Map Routing Machine (OSRM) using the Open Street Map North America data extract as of June 7, 2017, which allows for the calculation of routing for cars, but not for public transportation, which we do not include in this analysis.

    Open Street Map is an open source, crowdsourced dataset, which means it is constantly updated by a group of dedicated volunteers, and therefore may have some errors, be dated in certain locations, and contain omissions. We use the routing engine to calculate the fastest driving distance and time with no traffic between each of the over 154 million pairs, using the OSRM /route/v1/driving endpoint. For more information on how these routes are calculated, see OSRM.

    For the full process to create the data yourself, see the Github repository at https://github.com/UI-Research/spark-osrm

     

    Data Available: All Census tract pairs within 150 miles of each other by straight line distance are included, using data from 2017 and 2010 Census tracts, per the documentation above.

    File Location: The entire dataset is available as a single file, at s3://lsdm-emr-util/lsdm-data/travel-times/drive_times.parquet (864MB)
